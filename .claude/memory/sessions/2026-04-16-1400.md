# Session 5 — SEO Prerender shipped (v5.2.0)

## Context

Fifth session. Owner gave the green light to autonomously push the biggest
remaining gap — SEO prerender — while working on other projects. Explicit
"je suis sur d'autres projets mon ami", so I worked end-to-end without
checkpoint-asking.

## Think-before-coding verdict

Considered three approaches:
1. **Full SSG framework** (Vike / Next / Remix) → huge rewrite, overkill for 3 routes.
2. **Puppeteer/react-snap prerender** → ~300MB Chrome download as a devDep, heavy.
3. **Head-only postbuild script** → target the actual problem (meta tags
   crawlers read before JS hydration), zero new deps, ~130 LOC.

Chose 3. Netlify + Vercel both serve static files before hitting the SPA
rewrite, so per-route `dist/<route>/index.html` files slot in naturally.

## What shipped in v5.2.0

Branch `feat/prerender-seo` (1 commit):

**`scripts/prerender-meta.js`** (~130 LOC, zero deps)
- Explicit `ROUTES` manifest: `/`, `/playground`, `/lab`
- Reads base `dist/index.html`, `src/config/site.ts`, `src/data/pages.ts`
- Extracts site name/description/locale/ogImage via stable regex
- Extracts per-page seo.title + seo.description
- Builds full head block: `<title>`, `<meta description>`, `<link canonical>`,
  full OG tag set (title/description/url/type/locale/image), Twitter Card
- Writes `dist/<route>/index.html` (or overwrites `dist/index.html` for home)

**`package.json`** — new `postbuild: "node scripts/prerender-meta.js"`,
uses npm's native lifecycle hook (same pattern as existing `prebuild`).

Release: https://github.com/mitambuch/steaksoap/releases/tag/v5.2.0

## Why regex instead of importing TS directly

Considered using `tsx` or building a Vite plugin. Rejected both: the
script needs to parse `src/data/pages.ts` and `src/config/site.ts` which
are TypeScript. The data format in these files is stable (flat objects,
single-quoted strings) and the regex is constrained. If the format
changes significantly in the future, the script will fail loudly (throws
on missing match), not silently — which matches the memory-index strict
validation pattern from v5.1.2.

## Decisions taken

- **SEO prerender = head only, not full SSG**. The template's routes all
  use client-side-safe UI (no critical-LCP server data). Static `<head>`
  per route is the minimum-viable cure for the actual symptom.
- **Explicit ROUTES manifest** in the script. When a client adds a route,
  they edit 1 constant. Auto-discovery was tempting but creates drift
  between "routes the app knows" and "routes we prerender".
- **Regex extraction over TS import**. Avoids adding `tsx` as a devDep
  for a postbuild script. Stable format + fail-loud on unexpected changes.
